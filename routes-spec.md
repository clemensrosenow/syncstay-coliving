# SyncStay Platform Routes Specification

This document outlines the core routes needed to support the SyncStay coliving platform, specifically focusing on the user journey from discovery to reservation, and managing the unique "pending" vs "committed" pod states.

## 1. Landing Page
**Path:** `/`

*   **Main Function:** Introduce the Pod-First concept and act as the entry point using the hardcoded location and timeframe constraints.
*   **Key UI Elements:**
    *   Hero section with a simplified search form (Location Dropdown, Month Dropdown).
    *   "How it Works" section explaining the "Gravity" matching and simulated pre-authorization.
    *   Simple feed/grid of featured Anchor Properties in predefined hubs.
*   **UX Enhancements:**
    *   Clear, static dropdowns instead of complex date-pickers or typeaheads to ensure rapid MVP delivery.
    *   High-contrast CTA to direct users quickly into the funneled search.


## 2. Search & Discovery
**Path:** `/search`

*   **Main Function:** Display Anchor Properties in the selected location/month, sorted strictly by **AI Pod Compatibility Score**. Empty properties appear at the bottom.
*   **Key UI Elements:**
    *   List/Grid layout of Property Cards (Complex interactive map views are excluded to save sprint time).
    *   Property cards featuring an image, flat-rate pricing, and a prominent "AI Match / Pod Gravity" explanation (e.g., "95% Match: Sarah and Alex are pending here. You share similar work hours.").
    *   Basic filter toggle if necessary (e.g., Hide full pods), avoiding complex multi-filters like dynamic price ranges.
*   **UX Enhancements:**
    *   LLM-generated hook text explicitly featured on the cards to drive engagement.
    *   Skeleton loaders for rapid page transitions while the FastAPI microservice ranks pods.
*   **URL Params:**
    *   `?location=` (List of Strings: e.g., ["lisbon", "bali"] - matching predefined hubs)
    *   `?month=` (List of Strings: e.g., ["2026-10", "2026-11"] - matching static calendar blocks)

## 3. Property Details (Anchor Selection)
**Path:** `/properties/[propertyId]`

*   **Main Function:** Show details of an Anchor Property and list the travelers already pending in its Pod.
*   **Key UI Elements:**
    *   Image gallery and basic property details (amenities, flat-rate price).
    *   **Pod Overview Block:** Details of existing pending members and the active user's AI compatibility synthesis generated by the LLM.
    *   Room selection and a dominant "Commit to Pod" CTA.
*   **UX Enhancements:**
    *   Simple UI state showing how many spots are left to trigger "LOCKED" status (min_occupancy threshold).
*   **URL Params:**
    *   `[propertyId]` (Path parameter)
    *   `?month=` (String, explicitly passed from search)

## 4. Checkout & Reservation (Simulated)
**Path:** `/checkout`

*   **Main Function:** Handle the simulated financial commitment, emphasizing the frictionless booking state machine without real payment gateways.
*   **Key UI Elements:**
    *   Booking summary (Property, Month, Flat-Rate Price).
    *   **Status Notice:** Explicit copy stating "You will only be charged if the Pod reaches minimum occupancy" (Simulated).
    *   A simple "Confirm Commitment" button (Replaces Stripe/PayPal forms, ensuring sprint feasibility).
*   **UX Enhancements:**
    *   Instantaneous feedback loop (success toast) updating user state to `PENDING` via Next.js state changes.
    *   Celebratory UI if this specific commitment triggers the Pod to shift to `LOCKED` (Match Successful).
*   **URL Params:**
    *   `?propertyId=` (String)
    *   `?month=` (String)

## 5. My Bookings Overview Dashboard
**Path:** `/bookings`

*   **Main Function:** Overview of the user's simulated travel plans, split strictly into `PENDING` and `LOCKED` (committed) states.
*   **Key UI Elements:**
    *   Tabbed navigation: "Pending" and "Committed".
    *   **Pending Tab Elements:** Progress bar showing spots needed for the Pod to lock.
    *   **Committed Tab Elements:** Mocked "Group Chat Unlocked" UI for locked pods.
*   **UX Enhancements:**
    *   Static UI state updates simulating real-time notifications.
    *   Clear, simple cards focusing purely on Pod status rather than bloated generic dashboard features.
*   **URL Params:**
    *   `?tab=` (String: "pending", "committed")
